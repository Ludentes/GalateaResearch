KEY FINDINGS
============

1. OFFLINE-FIRST IS HARD
   - Firebase & Supabase both fail offline session restoration
   - No provider handles it well
   - Solution: Custom JWT + token caching + validation

2. DEVICE AUTH ≠ USER AUTH
   - Kiosk/player need device-level credentials (API Key, Service Account)
   - Not OAuth (which is for users)
   - Admin dashboard needs user login (OAuth)

3. TOKEN STORAGE IS CRITICAL
   - AsyncStorage = compromised in minutes on rooted device
   - Use Keychain (iOS) or Keystore (Android)
   - Access tokens: memory only
   - Refresh tokens: secure storage only

4. PATTERNS RANK BY SECURITY (⭐ scale)
   ⭐⭐⭐⭐⭐ OAuth PKCE + MFA (most secure)
   ⭐⭐⭐⭐ JWT + Token Rotation (good)
   ⭐⭐⭐⭐ Magic Link (good)
   ⭐⭐⭐ Device Service Account (medium, but offline-friendly)
   ⭐⭐⭐⭐ Session-Based (good, but mobile-unfriendly)

RECOMMENDATIONS BY USE CASE
============================

GUIDE KIOSK (Tablet, guide-controlled, offline WiFi)
─────────────────────────────────────────────────────
Pattern:        Device Service Account
Security:       ⭐⭐⭐ Medium (but device is trusted)
Implementation: 2-3 days
Offline:        ✓ Yes (with request queue)
Code:           In guides/MOBILE_AUTH_FOR_GALATEA.md, Use Case 1

Flow:
  1. First boot: Device registers, gets API key
  2. Credentials stored in Keychain
  3. All requests use device ID + API key
  4. Offline: Queue requests, sync when online
  5. Simple, no user login screen


PLAYER KIOSK (24/7 video loop, headless, automated)
────────────────────────────────────────────────────
Pattern:        JWT + Scheduled Refresh
Security:       ⭐⭐⭐⭐ Good
Implementation: 2-3 days
Offline:        ⭐⭐ Partial (token cache)
Code:           In guides/MOBILE_AUTH_FOR_GALATEA.md, Use Case 2

Flow:
  1. Initial auth: Get access + refresh tokens
  2. Refresh token stored in Keychain
  3. Scheduled refresh every 1 hour (prevents token expiry)
  4. Monitor failures remotely
  5. Graceful fallback if refresh fails
  6. Content fetch uses access token


ADMIN DASHBOARD (Mobile version, user auth, occasional use)
─────────────────────────────────────────────────────────
Pattern:        OAuth PKCE + Biometric Unlock
Security:       ⭐⭐⭐⭐⭐ Excellent
Implementation: 3-5 days
Offline:        ✗ No (user auth required)
Code:           In guides/MOBILE_AUTH_FOR_GALATEA.md, Use Case 3

Flow:
  1. First login: OAuth browser flow (no password)
  2. Access token in memory, refresh token in Keychain
  3. On app foreground: Biometric unlock (not re-login)
  4. Auto-refresh before expiry
  5. 401 triggers refresh + retry
  6. Biometric adds UX without sacrificing security

CRITICAL SECURITY CHECKLIST
============================

TOKEN STORAGE
  [ ] Never use AsyncStorage
  [ ] Access tokens: Memory only
  [ ] Refresh tokens: Keychain/Keystore
  [ ] Clear all tokens on logout

TOKEN LIFECYCLE
  [ ] Access tokens short-lived (15 min)
  [ ] Refresh tokens long-lived (7-30 days)
  [ ] Automatic refresh on 401 status
  [ ] Token expiry validated before use
  [ ] New refresh token on each refresh

API SECURITY
  [ ] HTTPS always (no HTTP)
  [ ] Authorization: Bearer header
  [ ] Token not in URL query params
  [ ] Token not logged to console

OFFLINE HANDLING
  [ ] Requests queued if offline
  [ ] Queue persists to disk
  [ ] Queue flushed on reconnect
  [ ] Stale data handled gracefully
  [ ] Conflict resolution defined

SESSION PERSISTENCE
  [ ] Session persists across app restart
  [ ] Logout clears credentials completely
  [ ] Biometric binding validated
  [ ] Device lock timeout set
  [ ] Session separate from user session

DECISION QUICK REFERENCE
=========================

Choose DEVICE SERVICE ACCOUNT if:
  ✓ Device auth (kiosk, player, hardware)
  ✓ Offline support needed
  ✓ Simple implementation
  ✓ Long-lived credentials OK
  ✗ NOT for user login

Choose JWT if:
  ✓ Full control needed
  ✓ Custom auth flow
  ✓ Offline-friendly (with caching)
  ✓ Stateless server

Choose OAUTH PKCE if:
  ✓ User authentication
  ✓ Social login desired
  ✓ Standard pattern
  ✓ Industry best practice

Choose MAGIC LINK if:
  ✓ Fast MVP
  ✓ Passwordless
  ✓ Simpler than OAuth
  ✗ Email required

Choose SESSION if:
  ✓ Web-first app
  ✓ Server-side state OK
  ✓ Scaling not concern
  ✗ Mobile unfriendly

COMMON PITFALLS TO AVOID
========================

❌ Store tokens in AsyncStorage
❌ Log tokens to console
❌ Hardcode secrets in app
❌ Assume WiFi always available
❌ Ignore offline scenarios
❌ Use weak refresh strategy
❌ Skip token expiry validation
❌ Forget to clear tokens on logout
❌ Misconfigure deep linking (silent failure)
❌ Ignore provider-specific quirks

✓ DO:
✓ Use Keychain/Keystore
✓ Validate token expiry locally
✓ Queue requests offline
✓ Log token errors (not the token itself)
✓ Rotate tokens periodically
✓ Test with network disabled
✓ Monitor refresh failures
✓ Handle 401 gracefully
